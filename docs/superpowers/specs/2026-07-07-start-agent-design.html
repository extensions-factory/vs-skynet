Start Agent + Codex Preflight — Design

E1 · F1.1 · one command to launch a codex session, gated by an install/sign-in preflight.

2026-07-07 US-1 start in one command US-2 detect missing / signed-out CLI MVP / v0.1

Overview

The interactive codex adapter is built and released — it can drive a codex TUI, harvest usage, and expose send()/dispose(). What is missing is a way to start one. The command palette has Send Task and Stop Agent, but nothing calls setActiveSession, so both always report "No running agent."

This feature is the first caller of the released adapter. It adds Skynet: Start Agent — one command that launches a codex session in a terminal and registers it as the active session — gated by a preflight that confirms codex is installed and signed in, with a clear fix when it is not. No adapter code changes.

Context & Assumptions

  • The adapter layer is released: codexAdapter.runInteractive(opts) returns an InteractiveSession; readiness (turn-0) runs inside it.
  • Requires an isolated CODEX_HOME. The adapter reads opts.configDir ?? process.env.CODEX_HOME and throws if neither is set. A real run needs a dedicated home, not the global account.
  • A1 — single active session (MVP). Parallel agents are E2. If a session is active, Start Agent refuses.
  • A2 — login-status surface (probed, codex-cli 0.142.5). codex login status exits 0 either way; the states are told apart by stdout (Logged in using ChatGPT vs Not logged in). codex --version fails to spawn when codex is off PATH.
  • A3 — preflight is a diagnostic, not orchestration. Child-process --version/login status do not violate the pty-only rule; CONSTITUTION.md §CLI-integration requires confirming installed + signed-in before use.
  • A4 — the "one turn result" skeleton is already covered by F1.3's Send Task; Start Agent only launches and reaches readiness.

Scope

Goals

  • A Skynet: Start Agent command that launches a codex session and registers it — no per-run prompts.
  • A preflight that detects not-installed / not-signed-in before opening a terminal, with an actionable fix.
  • Config via settings (skynet.codex.home, .model, .sandbox) with silent defaults.

Non-Goals

  • No standalone Check Codex command — preflight lives inside Start Agent.
  • No session replace / multi-session / fleet (E2). No model/sandbox quick-pick. No adapter changes.
  • No automated recovery, no Windows-specific preflight tuning.

User Stories

US-1 — Start an agent in one command (P1)

As a developer, I want to start a coding agent in a VSCode terminal with one command, so that I can begin without any per-run setup.

  • GIVEN codex installed + signed in, a folder open, a home configured, WHEN I run Start Agent, THEN a session launches, becomes active (setActiveSession), and the terminal is revealed.
  • GIVEN a session is already active, WHEN I run Start Agent, THEN it refuses and launches nothing.
  • GIVEN no folder open / no home configured, WHEN I run Start Agent, THEN it aborts with a fix-it message and no terminal opens.

US-2 — Detect a missing or signed-out CLI (P1)

As a developer, I want Skynet to tell me when codex is not installed or not signed in, so that I know how to fix it instead of staring at a broken terminal.

  • GIVEN codex off PATH, THEN preflight reports "not installed" with a hint; runInteractive is never called.
  • GIVEN login status reports Not logged in, THEN preflight reports "not signed in" with Run: codex login; runInteractive is never called.
  • GIVEN login status reports Logged in, THEN preflight returns ok with the parsed version and launch proceeds.

Approach

Two small, pure, dependency-injected units wired in the composition root — the same shape as the repo's task-handoff.ts / notify-awaiting-input.ts (no vscode in the logic; the root injects window, config, adapter, and a child-process runner). preflight.ts takes an injected run so it is testable with no real spawn; start-agent.ts takes a deps bag so it is testable with no real terminal. The preflight parses stdout (both states exit 0). Config resolves skynet.codex.home → CODEX_HOME → error.

AlternativeWhy rejected
Env-var-only homeForces launching VSCode with the env set; a setting is discoverable and still falls back to env.
Prompt / quick-pick a home on startUI for a first-run step a one-time setting covers; breaks "one command."

Design

Architecture

extension.ts (composition root)
  registers "skynet.startAgent"
  injects: getConfig, getWorkspaceCwd, checkCodex (bound to real execFile runner),
           runInteractive, setActiveSession, getActiveSession, window, genWorkerId
        │
        ▼
start-agent.ts  startAgentCommand(deps)
  1. active session already set?     → refuse, return
  2. resolve cwd                      → none → error, return
  3. resolve codexHome (setting→env)  → none → error, return
  4. checkCodex(codexHome)            → not ok → show fix, return   ◄─ no terminal yet
  5. build InteractiveOpts from settings
  6. runInteractive(opts) → session
  7. setActiveSession(session); reveal terminal
        │
        ▼
preflight.ts  checkCodex(run, codexHome): Promise<PreflightResult>
  run("codex", ["--version"])                    spawn fail → not-installed
  run("codex", ["login","status"], {CODEX_HOME}) stdout has "Not logged in" → not-signed-in
                                                  else → ok(version)

Components & Interfaces

src/adapters/codex/preflight.ts — injected Run = (cmd, args, env?) => Promise<{code,stdout,stderr}>; checkCodex(run, codexHome) → { ok:true, version } | { ok:false, reason:"not-installed"|"not-signed-in", message }. No vscode, no child_process import.

src/commands/start-agent.ts — startAgentCommand(deps); deps = { getActiveSession, setActiveSession, getWorkspaceCwd, getConfig, checkCodex, runInteractive, genWorkerId, window }. Structural window only (mirrors HandoffWindow). Terminal reveal is delegated to the adapter's Doorbell.

src/extension.ts — a real Run over node:child_process execFile (ENOENT → non-spawn result the preflight reads as not-installed); registers skynet.startAgent; generates workerId (codex-<short-unique>).

package.json — command skynet.startAgent ("Start Agent", Skynet); settings skynet.codex.home, .model, .sandbox (enum, default workspace-write); activation onCommand:skynet.startAgent.

Config → InteractiveOpts

{ cwd:       getWorkspaceCwd(),
  workerId:  genWorkerId(),
  configDir: home,                       // setting → env; required
  model:     config.model || undefined,  // omit → codex default
  sandbox:   config.sandbox ?? "workspace-write" }

Error handling & edge cases

  • Already active / no workspace / no home / preflight-fail → distinct showErrorMessage, return before any terminal opens and before runInteractive.
  • runInteractive throws (e.g. readiness timeout) → catch, show error, leave active session unset.
  • login status in an unexpected format → absence of Not logged in is treated as ok (fail-open; the adapter's readiness turn-0 catches a truly broken session as a timeout).
  • Empty/whitespace skynet.codex.home → treated as unset → env fallback.

Testing Strategy

All vitest, colocated, injected deps — no real spawn, no real terminal (matches task-handoff.test.ts).

Success Criteria

  • SC-1: With codex installed/signed-in, a folder open, a home configured, one Start Agent yields a running active session that Send Task can drive — no per-run prompts.
  • SC-2: Each failure class (not installed / not signed in / no home / no folder) gives a distinct actionable message and never opens a terminal or calls runInteractive.
  • SC-3: All units covered by injected-dependency unit tests under pnpm run test:unit; no real codex install or terminal required.
  • SC-4: No new provider-SDK/API dependency, no adapter change; the preflight is the only new child-process use, confined to install/auth diagnostics.
