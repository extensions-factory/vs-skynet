Task Hand-off — Design

E1 · F1.3 · hand a task to the running agent, and stop it — from the command palette.

2026-07-06 US-1 send a task US-2 stop the agent MVP / v0.1

Overview

A developer running a Skynet agent needs to hand it a follow-up task without leaving the editor, and stop it at any moment. This is a thin command layer over the already-built interactive session. Its central decision is a framing one:

Hand-off is a channel, not a UI. The channel is InteractiveSession.send(prompt), which already exists. The command palette is merely the MVP trigger; at E6 the Orchestrator drives the same channel. Naming the seam now gives E6 an obvious plug-in point and keeps "who talks to the worker" answerable in one sentence.
Command palette (human, MVP)  ─┐
Orchestrator   (machine, E6)  ─┴─→ InteractiveSession.send(prompt) → TurnResult
                                   InteractiveSession.dispose()    → stopped

Context (probed 2026-07-06)

  • InteractiveSession already exposes send(prompt) → TurnResult, status, sessionId, dispose(). F1.1/F1.2 built and merged this layer.
  • Transition table: send is legal only from ready and awaiting-input (both → busy). From busy there is no send edge — it throws illegal transition: busy -/send. send() also throws "session already completed" when terminal.
  • dispose() (kill terminal + close mailbox + machine stopped) is the only stop mechanism — there is no mid-turn interrupt in the code.
  • src/extension.ts is still the scaffold (skynet.helloWorld); no real command wiring or session-holding exists yet.
Assumptions. A1 — single active session at MVP (parallel is E2). A2 — F1.1 owns starting and holding the session; because extension.ts is still a scaffold, F1.3 may establish the single-session holder (getActiveSession()) if F1.1 hasn't landed it. A3 — stop = dispose(), a hard stop; a stopped session is not resumable. (All confirmed with the product owner.)

Scope

  • In: two thin commands — skynet.sendTask and skynet.stopAgent — that resolve the single active session and delegate to send() / dispose(), with status guards.
  • Out: multi-session picker (E2) · orchestrator routing (E6) · mid-turn interrupt · prompt queueing while busy · resume after stop. All YAGNI or later-epic.

User Stories

US-1 — Send a task to the running agent (P1)

As a developer, I want to send a task/prompt to the running agent from the command palette, so that I can delegate work without switching context.

  • GIVEN ready/awaiting-input, WHEN I run skynet.sendTask and enter a prompt, THEN session.send(prompt) is called and its TurnResult.summary is surfaced.
  • GIVEN busy, WHEN I run it, THEN send() is not called and I see a "busy" message.
  • GIVEN no session, or a terminal session, WHEN I run it, THEN I see a "no running agent" message; nothing is sent.
  • GIVEN the input box is dismissed / empty, WHEN I cancel, THEN nothing is sent.

US-2 — Stop the running agent (P1)

As a developer, I want to stop an agent at any time, so that I always stay in control.

  • GIVEN any non-terminal state, WHEN I run skynet.stopAgent, THEN session.dispose() is called and status becomes stopped.
  • GIVEN no session, or an already-terminal session, WHEN I run it, THEN I see a message and dispose() is not called again.

Approach

Add two thin VSCode commands that resolve the single active session from the composition root and delegate to the existing InteractiveSession methods. No new core class: the hand-off seam is session.send() itself — already present, already the contract the Orchestrator will call. Handlers guard on session.status before acting, so the state machine's illegal-transition throw never fires on the normal path.

AlternativeWhy rejected
A TaskDispatcher.handOff() core wrapper for both callersOne caller today; the seam already exists as send(). A wrapper is speculative until E6 needs routing/logging. YAGNI.
MVP mini-orchestrator, all hand-off routed through itPulls E6 (v0.5) into v0.1. The direct command path is the honest MVP; the seam framing already reserves the orchestrator's future entry.
Keep "command palette" as the worker-comms conceptConflicts with the ARCHITECTURE Mediator rule ("workers never talk directly; the orchestrator routes hand-offs"). Reframing to a channel resolves it.
Stop = mid-turn interrupt, session survivesNo interrupt in the code; needs a new transport sequence + transitions. Deferred; dispose is the MVP stop.
Queue a prompt sent while busyAdds buffering + ordering for a case the user can just retry. YAGNI.

Design

Components

  • Active-session accessor — a single getActiveSession(): InteractiveSession | undefined in the composition root. Returns the one running session or undefined. Used by both handlers.
  • skynet.sendTask handler — resolve session; absent/terminal → message; busy → message; else showInputBox, and on non-empty input call session.send(prompt), await TurnResult, show summary.
  • skynet.stopAgent handler — resolve session; absent/terminal → message; else session.dispose().
  • Both declared in package.json contributes.commands and pushed to context.subscriptions in activate(). Handlers live at the edge (they touch vscode); the session + state machine stay pure core, untouched.

Guards & error handling

  • Status is checked before every send()/dispose(), so illegal transition / session already completed throws aren't reachable on the normal path.
  • send() is still wrapped in try/catch; a thrown error, or a TurnResult of error/timeout/crashed, is surfaced via a message. No unhandled rejection escapes the handler.
  • Terminal-state check makes dispose() effectively idempotent (the machine also no-ops once terminal).

Edge cases

  • Input box dismissed / empty → no send, no message noise.
  • ready → busy race between the status check and the send → send() throws; the try/catch turns it into a "busy, try again" message. Accepted; no queue.
  • Stop while launching → dispose is legal from launching → stopped.
  • Stop twice quickly → first disposes; second sees terminal status → message, no double dispose.

Testing

Unit tests (vitest), no extension host — handlers take the active-session accessor and run against a fake InteractiveSession (stub send/dispose/status) with window.showInputBox/showInformationMessage mocked (existing src/test-utils/vscode-mock.ts pattern).

  • US-1: ready + prompt → send once, summary shown; busy → no send + message; no/terminal session → no send + message; empty input → no send.
  • US-2: non-terminal → dispose once, status stopped; terminal/no session → no dispose + message.
  • Error path: fake send throwing → caught, message shown, no rejection.

Success criteria

  • SC-1 — a developer sends a prompt from the palette and sees the turn summary (US-1 tests pass).
  • SC-2 — a developer stops the agent and status becomes stopped (US-2 tests pass).
  • SC-3 — no handler can trigger an illegal transition/session already completed throw on the documented paths (guard tests).
  • SC-4 — the hand-off seam is InteractiveSession.send(prompt) with zero new core classes (diff inspection: edge-only changes).
