Interactive Codex Adapter — Design (canonical frame)

Adapters · drive codex as a live interactive TUI in a VSCode terminal — the only run mode (pty-only, no headless exec).

Goal

Drive codex as a live interactive TUI session inside a VSCode terminal. The orchestrator pastes prompts into the terminal and reads results out of files the agent writes. This buys multi-turn steering (paused/resumed turns toward one task completion) + pause/resume in one session and a human-like execution surface, while keeping parsing robust.

The only run mode. Per CONSTITUTION.md (pty/terminal only, no headless flags, no provider SDK), there is no codex exec --json path. A "fast task" is simply a single-turn mailbox session: one inbox turn, the agent returns done, dispose. Multi-turn and single-turn are the same mechanism at different turn counts.

Canonical frame for interactive mode across all three CLIs. The claude and agy skeletons are not yet ported to active/; only codex is in scope here.

Depends on (prerequisite)

active/ is currently a bare scaffold (only src/extension.ts); the base adapter layer this spec builds on does not exist yet and must land first. Reconciled with the constitution, the base layer is types + classifier only — no headless one-shot adapter:

  • Shared adapter types — src/adapters/types.ts: ErrorClass, WorkerEvent, WorkerUsage ({ inputTokens, outputTokens, cachedInputTokens?, cacheWriteTokens?, reasoningTokens?, costUsd? }), WorkerResult, RunOpts, WorkerRun, AgentAdapter.
  • Error classifier — src/adapters/classify.ts: pure classifyError(text): ErrorClass.
Deliberately no runCodex / codex exec --json dependency — the /6 one-shot adapter (6/…/2026-06-29-codex-adapter-design.md) is superseded by single-turn interactive mode. Hard prerequisite: without the two symbols above, interactive mode does not compile.

Why this shape (recorded decisions)

  • pty/terminal only (constitution). No headless exec, no provider SDK. Mailbox-over-a-real-terminal is the only mechanism; single-turn is the fast path, not a separate headless code path.
  • Ban risk is precautionary only (no observed bans). So we build no anti-ban machinery.
  • Control is the real win: multi-turn, pause/resume, live human takeover.
  • Public VSCode API cannot read a full-screen TUI's output. onDidWriteTerminalData is permanently proposed; no screen-buffer API. → Output flows through files, never screen scraping.
  • No node-pty. Native ABI must match VSCode's Electron. → Use VSCode's own createTerminal.
  • Session metadata is read from the CLI's own rollout file, not the screen.
  • Protocol never touches the project's real instruction file. It lives in the disposable, gitignored mailbox dir and is delivered by the readiness ping.

codex (Rust/ratatui TUI). Sources: developers.openai.com/codex, github.com/openai/codex, and the prior worktree's terminal-probe.test.ts (TERMINAL_PROBE=1, run 2026-07-01 against a real codex-cli 0.142.4 install). This port trusts that evidence rather than re-probing.

Launch interactive

codex -C <cwd> -m <model> -s workspace-write -a never
  -c disable_paste_burst=true
  -c 'tui.keymap.composer.submit="tab"'
  -c 'tui.keymap.composer.queue="ctrl-q"'
VERIFIED (prior worktree) — the probe launches with this exact -a never argv and completes a two-turn pause/resume/done cycle with no stuck approval prompt. -a never is confirmed auto-approve within sandbox, not deny everything. Flags are global (shared by codex, resume).

Session rollout file

$CODEX_HOME/sessions/YYYY/MM/DD/rollout-<ts>-<uuid>.jsonl (default ~/.codex). JSONL of RolloutLine envelopes, written incrementally:

  • session_meta → conversation/session id, cwd, cli version, git info.
  • event_msg with payload.type === "token_count" → cumulative usage in payload.info.total_token_usage (input_tokens, cached_input_tokens, output_tokens, reasoning_output_tokens, total_tokens); last_token_usage; model_context_window; payload.rate_limits. Per-turn = current cumulative − previous.

Resume, isolation, submit key

  • Resume: codex resume <id> / --last / /resume. Native resume is our crash-recovery fallback.
  • CODEX_HOME relocates the whole home (config + sessions/), so per-account isolation (via opts.configDir) moves the rollout files too.
  • /status prints session id/model/cwd/permissions/account/limits; diagnostic only.
  • Submit key: injection is paste-like; both \r and kitty [13u land as newlines. Verified path: launch with disable_paste_burst=true, composer.submit="tab", composer.queue="ctrl-q", submit with sendSequence("\t").

Architecture (canonical — shared by all three CLIs)

CLI-agnostic core in src/adapters/interactive/; each CLI supplies a profile.

   + inbox/turn-0.md (readiness)

· timeout: no outbox within T  → status 'timeout'
· poll process group / recursive descendants: no codex → status 'crashed'
· onDidCloseTerminal / exitStatus → terminal died

Components

  1. TerminalSession — wraps vscode.window.createTerminal({ name, cwd, env }). Captures shell PID via terminal.processId. Owns disposal + onDidCloseTerminal.
  2. Mailbox — per-run dir <cwd>/.skynet/<workerId>/{inbox,outbox}/ plus protocol.md. Resolves the turn by polling outbox/turn-N.json (no FileSystemWatcher). Ensures .skynet/ is gitignored. dispose() removes the whole <workerId> dir. The poll (~500ms, same timeout as the turn) is the only detection mechanism and doubles as the parse-error retry: on ENOENT or a JSON parse error it keeps polling until valid or timeout.
  3. Doorbell — terminal.show(false) → sendText(pingLine, false) → sendSequence(profile.submitSequence). The ping is tiny so it dodges large-paste corruption.
  4. Protocol file — the mailbox protocol is written once to .skynet/<workerId>/protocol.md at session start and delivered by the turn-0 readiness ping. The project's real instruction file (AGENTS.md, …) is never touched — no marker-block append, no teardown, no crash-leftover in a tracked file. The continuous TUI keeps the contract in context; each inbox/turn-N.md also closes with a one-line reminder as belt-and-suspenders. (ponytail: deletes the /6 instruction-file bootstrap/teardown component and its marker machinery entirely.)
  5. SessionHarvester — newest rollout-*.jsonl under profile.sessionDir(configDir) → { sessionId, usage, rateLimits? }. Read each turn and at dispose.
  6. Optional SessionInfoProbe — asks the agent to write outbox/session-info.json for CLIs without a useful transcript. N/A for codex.
  7. InteractiveSession (orchestrator) — the state machine driving turns.

Public shape

type TurnResult =
  | { status: "paused";  summary: string }
  | { status: "done";    summary: string; usage?: WorkerUsage; filesTouched?: string[] }
  | { status: "error";   reason: string; errorClass?: ErrorClass }
  | { status: "timeout" }
  | { status: "crashed" };

interface InteractiveSession extends AsyncIterable<WorkerEvent> {
  send(prompt: string): Promise<TurnResult>;
  readonly sessionId: Promise<string | undefined>;
  dispose(): Promise<void>;
}

interface AgentAdapter {                               // base layer — no headless run()
  readonly id: "codex" | "claude" | "agy";
  runInteractive(opts: InteractiveOpts): Promise<InteractiveSession>;  // multi- or single-turn
  // A "fast task" = start, send one turn, expect done, dispose. Not a separate path.
}

After a turn resolves done, the session is complete: further send() calls reject and dispose() is the only valid next call. TurnResult maps to WorkerResult only at the adapter boundary: done→success, error→failed, timeout→failed(transport), crashed→failed(terminal). paused keeps the session open.

Enrichment note (ordering contract). The started event carries sessionId, but that id comes from the asynchronous rollout harvest, not synchronously at launch. started is emitted once the first harvest yields a sessionId (or empty if unresolved by the first turn's completion — the readonly sessionId promise is authoritative). Consumers must not assume started precedes the first send(); the iterator is passive.

Protocol contract (delivered via .skynet/<workerId>/protocol.md)

Written once at session start; the turn-0 ping tells the agent to read it. For each inbox/turn-N.md: do the work, then write outbox/turn-N.json before you stop, matching the same N — readiness (turn 0) {"status":"paused","summary":"ready"} · {"status":"paused","summary":"…"} · {"status":"done","summary":"…","filesTouched":["…"]} · {"status":"error","reason":"…"}. Never delete inbox files. Write the outbox in a single operation (tmp + rename) as the last action of the turn. The mailbox retry stays mandatory because the agent may ignore this.

The outbox existence is the turn boundary; its status decides the next move. For Codex, usage/session-id never come from the agent — they come from the rollout harvest.

Readiness handshake & sad path

  • Readiness is turn-0, separate from real work. A synthetic turn-0 only asks the agent to read protocol.md and write outbox/turn-0.json = {"status":"paused","summary":"ready"}. No outbox/turn-0.json within readyTimeoutMs (30s) → re-send the turn-0 ping once before failing. Safe because turn-0 does no real work — re-reading + re-writing ready is idempotent. A ~1.5s pre-ping delay reduces the sendText startup race.
  • Real-work turns (N ≥ 1) are never resent. The /6 spec used turn-1 as both readiness probe and first real task, so a first task slower than readyTimeoutMs got re-pinged and could re-execute (duplicate work). Splitting readiness into an idempotent turn-0 removes that hazard; real turns' only guard is turnTimeoutMs.
  • Turn timeout (turnTimeoutMs, 5 min): no outbox → timeout. Sole guard for real-work turns.
  • Crash: poll the process group / walk descendants of terminal.processId (macOS/Linux, ~3s); no codex descendant → crashed. Best-effort.
  • Terminal death: onDidCloseTerminal / exitStatus → crashed.
  • Recovery via codex resume is out of scope for v1.

Error classification

Reuse the base-layer pure classifyError(text). Inputs: agent-written error.reason plus any stderr from terminal close. A real 429 mid-TUI may kill the session before error.json is written; then we report crashed, and a later fallback US classifies from the rollout/last-known state.
